# 실행 기준
- DB/Redis/App은 프로젝트 루트 docker-compose.yml 기준으로 실행
- infra는 prometheus/grafana/k6 전용으로 실행
# 부하 테스트 세부 환경 설정
- 저렴한 환경 부하 테스트 : 문제 탐지(Probe)
- 운영 환경 부하 테스트 : 수치 확정, 운영 유사(Prodlike)

# 컨테이너별 자원 제한

## Probe(문제 탐지용) 자원 제한
- prometheus: cpus: 0.25, memory: 512M
- grafana: cpus: 0.25, memory: 512M
- k6: cpus: 0.50, memory: 512M
- Probe 총합(대략): CPU 1.0 / RAM 1.5GB

## Prodlike(운영 유사) 자원 제한
- prometheus: cpus: 0.75, memory: 1.5G
- grafana: cpus: 0.50, memory: 1G
- k6: cpus: 1.00, memory: 1G
- Prodlike 총합(대략): CPU 2.25 / RAM 3.5GB

# 공통 규칙(모든 시나리오에 적용)
## 인증
- 기본 : Authorization: Bearer {apiKey} {accessToken}
- 토큰 공급 : k6/data/tokens.txt (한 줄에 apiKey accessToken)

## 판정 PASS/FAIL
각 시나리오는 느낌이 아니고, 불변조건/수치로 FAIL 출력
- 정합성 : ex) currentHighestBid == MAX(bids)
- 안정성 : 오류율(5xx) 증가
- 성능 : p95/p99 임계치 초과
- DB : lock wait, slow query 증가

# Google sre 표준 지표 + 추천 지표
- Google sre 표준 지표 : Latency / Error / Traffic / Saturation
## Latency - 지연시간
- p95
- p99

k6 지표
- k6_http_req_duration_bucket

grafana 쿼리
```
histogram_quantile(
  0.95,
  sum(rate(k6_http_req_duration_bucket[1m])) by (le)
)
```
```
histogram_quantile(
  0.99,
  sum(rate(k6_http_req_duration_bucket[1m])) by (le)
)
```
해석 기준
- p95 < 500ms : ok
- p99 > 1s : 병목 존재
- p99 급등 : 락, 커넥션 풀, GC 의심

## Error - 실패율
- 에러율 %
- http status 분포

k6 지표
- k6_http_reqs
- k6_http_req_failed

grafana 쿼리
```
sum(rate(k6_http_req_failed[1m]))
/
sum(rate(k6_http_reqs[1m]))
```

해석 기준
- 에러율 > 1% - 사용자 불만 존재
- 에러율, p95 급등이 동시 발생 - 서버 한계 초과

## Traffic - 트래픽
- RPS. request per second
- VUS

k6 지표
- k6_http_reqs
- k6_vus
- k6_vus_max

grafana 쿼리 (rps)
```
sum(rate(k6_http_reqs[1m]))
```

해석 기준
- rps 원인
- p95/p99 결과
- rps 증가 - p95 증가 - 에러 증가

## saturation - 포화 상태
k6가 아니고, 서버/DB 메트릭.
### cpu
- process_cpu_usage
- container_cpu_usage_seconds_total

해석 기준
- cpu 80% 이상 지속 - scale 필요

### memory/gc
JVM
- jvm_memory_used_bytes
- jvm_gc_pause_seconds_sum
MySQL
- mysql_global_status_buffer_pool_pages_free

해석 기준
- p95 증가 - gc pause 증가 - 메모리 압박

### db connection
- hikaricp_connections_active
- hikaricp_connections_pending

해석 기준
- pending > 0 : 병목
- 이 상태에서 p99 튀면 DB 문제

## k6 자체 지표
- http_req_waiting (TTFB)
- http_req_sending
- http_req_receiving

해석 기준
- waiting만 길 때 - 서버/db
- sending/receiving 길 때 - 네트워크/응답 크기
